Resumen ejecutivo
Visión general en una página

Madcry Studio es la galería online y tienda directa de la artista Susana Madriz. La plataforma permite mostrar el catálogo de obra, gestionar la venta de piezas únicas (1/1) con cobro a tarjeta, recoger comisiones a medida a través de un formulario de contacto y administrar todo el contenido desde un panel privado.

El sitio se construye con Astro 5 en modo estático (SSG): cada despliegue genera HTML pre-renderizado servido desde una CDN global. La interactividad puntual (panel admin, botón de compra) se hidrata con Vue 3. El backend transaccional vive en Supabase (PostgreSQL gestionado, autenticación, almacenamiento de imágenes y funciones serverless), mientras que el cobro lo realiza Stripe en modo Checkout, sin que datos de tarjeta lleguen nunca al servidor de Madcry.

Principios rectores

  • Privilegio mínimo. Cada componente accede sólo a lo que necesita. La clave anónima del cliente no puede leer pedidos. Las credenciales potentes viven exclusivamente en funciones serverless.
  • Defensa en profundidad. Múltiples capas (CDN, CSP, RLS, JWT, validación servidor, captcha futuro) impiden que el fallo de una sola comprometa el sistema.
  • Estático por defecto. Lo que se puede pre-renderizar, se pre-renderiza. Reduce superficie de ataque y coste a la vez.
  • Auditable. Todo el código está versionado en Git. Cada despliegue es trazable. Las funciones serverless registran logs estructurados.
●

Resultado de la auditoría interna (mayo 2026): sin vulnerabilidades críticas. Mejoras menores de SEO y endurecimiento de CSP en cola para próxima iteración. Detalle en sección 06.

Arquitectura
Visión por capas y separación de responsabilidades

La plataforma sigue una arquitectura JAMstack con backend gestionado: el cliente es estático, los servicios sensibles viven detrás de funciones serverless aisladas, y los proveedores externos (Stripe, Resend) se invocan exclusivamente desde el lado servidor con credenciales nunca expuestas al navegador.

Diagrama de capas

Capa 1 · Cliente
Navegador
  • Astro HTML estático
  • Vue 3 (islas)
  • Sólo claves públicas
  • CSP estricta + HSTS
Capa 2 · Borde
CDN + Funciones
  • Firebase Hosting
  • Edge Functions (Deno)
  • JWT + CORS allowlist
  • Rate-limit por IP
Capa 3 · Núcleo
Datos y pago
  • Postgres con RLS
  • Stripe (PCI-DSS L1)
  • Resend (email)
  • Storage + bucket auth
▲ ▼ comunicación cifrada TLS 1.3

Flujos críticos
Compra de obra, formulario de contacto, publicación admin

Flujo 1 · Compra de una pieza

[1] Cliente entra en /product/<slug> (HTML estático servido por CDN) [2] Click en "Adquirir Obra" [3] Navegador llama a create-checkout-session (Edge Function) → CORS verificado contra allowlist → Lee precio FRESCO de la BD (no confía en el cliente) → Verifica status='available' y categoría no es por encargo [4] Crea Stripe Checkout Session con price_data inline [5] Pre-crea fila en orders con status='pending' [6] Devuelve URL de Stripe; navegador redirige al cliente [7] Cliente paga en dominio Stripe (PCI-DSS) [8] Stripe llama al webhook stripe-webhook → Firma HMAC verificada con STRIPE_WEBHOOK_SECRET → Marca pedido como 'paid', producto como 'sold' → Envía email cliente (Resend) y email admin [9] Cliente vuelve a /success?session_id=cs_... [10] Página llama a verify-session → Confirma con Stripe que el cobro está pagado → Devuelve email enmascarado (no completo)
✓

Garantía técnica: el precio nunca puede ser manipulado desde el navegador. La Edge Function relee el valor de Postgres en cada checkout. Si un atacante intentara inyectar otro price_cents, el servidor lo descartaría.

Flujo 2 · Mensaje de contacto / comisión

[1] Visitante envía formulario en /contact [2] Edge Function send-contact-email ejecuta: → Honeypot anti-bots (campo invisible) → Validación email + límite de 5000 caracteres → Rate-limit: 3 mensajes/hora por email · 10/hora por IP → Inserta en tabla messages → Notifica a info@madcry.com vía Resend [3] Confirmación al usuario (200 OK con CORS allowlist)

Seguridad — capas y controles
Defensa en profundidad sobre la plataforma

Los controles se agrupan en cuatro capas concéntricas: red, transporte, aplicación y datos. Un fallo en una capa no compromete las siguientes.

Capa de red y transporte

01
HTTPS forzado
Todo el tráfico circula sobre TLS 1.3. HTTP redirige a HTTPS por defecto en Firebase Hosting.
02
HSTS preload
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload — el navegador rechaza HTTP futuro durante 1 año.
03
CDN distribuida
Mitigación implícita de DDoS volumétrico vía red global de Firebase (Cloud CDN de Google).

Capa de aplicación · cabeceras

04
Content-Security-Policy
Política estricta: default-src 'self'; conexiones limitadas a Supabase/Stripe; frame-ancestors 'none' impide clickjacking.
05
X-Frame-Options DENY
Refuerzo doble anti-clickjacking para navegadores antiguos.
06
X-Content-Type-Options
nosniff — el navegador no interpreta MIME types incorrectos.
07
Referrer-Policy
strict-origin-when-cross-origin — no fuga de paths privados a terceros.
08
Permissions-Policy
Bloquea cámara, micrófono, geolocalización; permite payment sólo a dominios de Stripe.
09
Cross-Origin-Opener-Policy
same-origin — aísla la pestaña principal de pop-ups maliciosos.

Capa de aplicación · API

10
CORS allowlist
Las funciones serverless sólo aceptan peticiones de madcry.com y entornos de desarrollo. Antes era abierto a todos los dominios.
11
Verificación JWT
Funciones admin (admin-product-sync, trigger-rebuild, send-shipping-email) exigen JWT válido + email exacto del admin.
12
Verificación de firma Stripe
El webhook valida HMAC-SHA256 con STRIPE_WEBHOOK_SECRET. Sin firma válida no se procesa el evento.
13
Rate-limit por IP y email
Formulario de contacto: máx. 3/h por email + 10/h por IP. Función rebuild: throttle 60 s.
14
Honeypot anti-bots
Campo invisible en el formulario. Bots lo rellenan, humanos no. Reduce ~95% del spam automatizado.
15
Sanitización de URLs de imagen
Antes de tocar Stripe / BD, se filtran esquemas javascript:, data:, vbscript:.
Madcry Studio · Documentación técnica Seguridad · 09 v1.0 · Mayo 2026

Capa de datos

16
Row-Level Security
Postgres aplica políticas a cada fila. La clave anon del cliente no puede leer pedidos ni mensajes ni borradores. Verificable por SQL.
17
Vista pública limitada
products_public excluye productos en borrador. Es la única superficie expuesta en build.
18
Aislamiento de service_role
La clave maestra que ignora RLS vive sólo dentro de Edge Functions, nunca en el navegador, nunca en logs.
19
Almacenamiento con MIME whitelist
Bucket products sólo acepta JPEG, PNG, WebP, AVIF. Tope de 10 MB por archivo.
20
Cifrado en reposo
Todos los datos en Postgres y Storage cifrados con AES-256 por la propia plataforma de Supabase.
21
Email enmascarado en respuestas públicas
verify-session devuelve sólo iniciales + dominio (su***@dominio.com) para evitar enumerar clientes con un session_id filtrado.
22
Soft-delete de productos
No se borran filas: pasan a status draft y el producto Stripe se archiva. Mantiene integridad referencial con pedidos históricos.

Pagos · separación de responsabilidades

▣

Madcry Studio nunca almacena ni recibe datos de tarjeta. El proceso de cobro ocurre íntegramente en infraestructura de Stripe (PCI-DSS Level 1, máximo nivel de cumplimiento). Madcry sólo recibe del webhook el resultado del pago, los datos de envío del cliente y el identificador opaco cs_… de la sesión.

Gestión de secretos y credenciales
Inventario, almacenamiento y procedimiento de rotación

Las credenciales se clasifican en dos grupos según su exposición:

Públicas (visibles en bundle del navegador)

Variable Origen Riesgo
PUBLIC_SUPABASE_URL Supabase Nulo · es URL pública del proyecto.
PUBLIC_SUPABASE_ANON_KEY Supabase Bajo · sólo permite lo que las políticas RLS permitan.
PUBLIC_STRIPE_PUBLISHABLE_KEY Stripe Nulo · diseñada para ir en cliente.

Privadas (solo en Edge Functions / GitHub Secrets)

Variable Almacenamiento Procedimiento de rotación
SUPABASE_SERVICE_ROLE_KEY Edge Functions secrets Dashboard Supabase → Settings → API → Reset.
STRIPE_SECRET_KEY Edge Functions secrets Stripe Dashboard → Developers → API Keys → Roll.
STRIPE_WEBHOOK_SECRET Edge Functions secrets Stripe Dashboard → Webhooks → Endpoint → Roll secret.
RESEND_API_KEY Edge Functions secrets Resend Dashboard → API Keys → Revoke + recreate.
SUPABASE_DB_PASSWORD Solo terminal puntual Supabase Dashboard → Database → Reset password.
GITHUB_TOKEN (deploy) Edge Functions secrets GitHub Settings → Developer settings → Tokens.
FIREBASE_SERVICE_ACCOUNT GitHub Actions secrets Firebase IAM → service account → New key.

Reglas internas de manejo

  • Nunca pegar una credencial privada en chat, email, ticket o screenshot.
  • Nunca persistirla en .env.local salvo si es estrictamente PUBLIC_*.
  • Si es necesaria de forma puntual, exportarla en la sesión de terminal y borrar histórico (history -c).
  • Rotar inmediatamente cualquier credencial que haya estado en un dispositivo perdido, repositorio público o canal compartido.
  • Toda rotación queda registrada con fecha en este documento (sección Anexo · Registro de rotaciones).
!

Política de mínimo necesario: el archivo .env.local que vive en la máquina de desarrollo contiene exclusivamente claves PUBLIC_*. Las claves potentes están en Supabase secrets — accesibles a la persona con login del proyecto, no al sistema de archivos del developer.

Operación, backups y continuidad
Cómo se despliega, se respalda y se recupera la plataforma

Despliegue

Cada commit en la rama main dispara el workflow deploy.yml de GitHub Actions, que ejecuta npm ci, astro build y un deploy a Firebase Hosting con la action oficial. Tiempo total medio: ~2-3 minutos. Es posible disparar el mismo workflow manualmente desde el panel admin (función trigger-rebuild).

Backups

Componente Frecuencia Retención Mecanismo
PostgreSQL Diaria automática 7 días (plan free)
30 días (plan Pro)		 Supabase Backups (PITR opcional)
Imágenes (Storage) Continuo Indefinido Replicación interna del proveedor
Código En cada commit Indefinido GitHub (clones distribuidos)
Versión desplegada En cada build 10 versiones Firebase Hosting · rollback en 1 click

Recuperación ante incidencias

Escenario Procedimiento RTO objetivo
Bug en producción Rollback en Firebase a versión anterior < 5 min
Pérdida de datos en BD Restauración de backup Supabase + replay de webhooks < 4 h
Compromiso de credencial Rotación según matriz de la sección 07 + auditoría de logs < 1 h
Caída del proveedor Comunicación al cliente · esperar SLA del proveedor Según SLA proveedor

Monitorización

Logs estructurados en cada Edge Function (console.log visible en Supabase Dashboard). Stripe envía email automático ante webhooks fallidos. Resend reporta entrega/rebote. Próxima iteración: integración con Sentry para errores de cliente y agregación de logs en una sola consola.
