Resumen ejecutivo
Visión general en una página

Madcry Studio es la galería online y tienda directa de la artista Susana Madriz. La plataforma permite mostrar el catálogo de obra, gestionar la venta de piezas únicas (1/1) con cobro a tarjeta, recoger comisiones a medida y administrar todo el contenido desde un panel privado.

El sitio se construye con Astro 5 en modo estático (SSG): cada despliegue genera HTML pre-renderizado servido desde una CDN global. La interactividad puntual se hidrata con Vue 3. El backend transaccional vive en Supabase (PostgreSQL gestionado, autenticación, almacenamiento y funciones serverless), mientras que el cobro lo realiza Stripe, sin que datos de tarjeta lleguen al servidor de Madcry.

Principios rectores

  • Privilegio mínimo. Cada componente accede sólo a lo que necesita. La clave anónima del cliente no puede leer pedidos. Las credenciales potentes viven exclusivamente en funciones serverless.
  • Defensa en profundidad. Múltiples capas (CDN, CSP, RLS, JWT, validación servidor) impiden que el fallo de una sola comprometa el sistema.
  • Estático por defecto. Lo que se puede pre-renderizar, se pre-renderiza. Reduce superficie de ataque y coste a la vez.
  • Auditable. Todo el código está versionado en Git. Cada despliegue es trazable. Las funciones serverless registran logs estructurados.
●

Resultado de la auditoría interna (mayo 2026): sin vulnerabilidades críticas. Mejoras menores de SEO y endurecimiento de CSP en cola. Detalle en sección 06.

Arquitectura
Visión por capas y separación de responsabilidades

La plataforma sigue una arquitectura JAMstack con backend gestionado: el cliente es estático, los servicios sensibles viven detrás de funciones serverless aisladas, y los proveedores externos se invocan desde el lado servidor con credenciales nunca expuestas al navegador.

Diagrama de capas

Capa 1 · Cliente
Navegador
  • Astro HTML estático
  • Vue 3 (islas)
  • Sólo claves públicas
  • CSP estricta + HSTS
Capa 2 · Borde
CDN + Funciones
  • Firebase Hosting
  • Edge Functions (Deno)
  • JWT + CORS allowlist
  • Rate-limit por IP
Capa 3 · Núcleo
Datos y pago
  • Postgres con RLS
  • Stripe (PCI-DSS L1)
  • Resend (email)
  • Storage + bucket auth
▲ ▼ comunicación cifrada TLS 1.3

Relación entre componentes

┌───────────┐ ┌──────────────┐ ┌────────────────┐ │ Visitante │───▶│ Firebase CDN │───▶│ HTML estático │ │ navegador │ │ (HTTPS+HSTS) │ │ de /dist/ │ └───────────┘ └──────────────┘ └────────────────┘ │ click "comprar" / submit form ▼ ┌──────────────────────────────────────────────────────────┐ │ Supabase Edge Functions (Deno · TLS · CORS allowlist) │ │ create-checkout · verify-session · stripe-webhook (HMAC) │ │ send-contact · admin-product-sync (JWT) · trigger-rebuild│ └──────────────────────────────────────────────────────────┘ │ │ │ ▼ ▼ ▼ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ Postgres │ │ Stripe │ │ Resend │ │ + RLS │ │ Checkout │ │ SMTP API │ │ + Storage │ │ + webhooks │ │ SPF/DKIM │ └────────────┘ └────────────┘ └────────────┘

Lectura clave: el navegador nunca habla directamente con Stripe ni con Postgres con privilegios elevados. Toda operación sensible pasa por una Edge Function que valida origen, autenticación y permisos.

Compra de obra, formulario de contacto, publicación admin

Flujo 1 · Compra de una pieza

[1] Cliente entra en /product/<slug> (HTML estático CDN) [2] Click en "Adquirir Obra" [3] Navegador llama a create-checkout-session → CORS verificado contra allowlist → Lee precio FRESCO de la BD → Verifica status='available' [4] Crea Stripe Checkout Session con price_data inline [5] Pre-crea fila en orders con status='pending' [6] Devuelve URL · navegador redirige [7] Cliente paga en dominio Stripe (PCI-DSS) [8] Stripe llama al webhook stripe-webhook → Firma HMAC verificada → Marca pedido 'paid', producto 'sold' → Envía emails (Resend) cliente y admin [9] Cliente vuelve a /success?session_id=cs_... [10] verify-session confirma con Stripe el pago → Devuelve email enmascarado (no completo)
✓

Garantía técnica: el precio nunca puede ser manipulado desde el navegador. La Edge Function relee el valor de Postgres en cada checkout.

Flujo 2 · Mensaje de contacto / comisión

[1] Visitante envía formulario en /contact [2] Edge Function send-contact-email: → Honeypot anti-bots → Validación email + 5000 chars max → Rate-limit: 3/h email · 10/h IP → Inserta en tabla messages → Notifica a info@madcry.com [3] Confirmación al usuario (CORS allowlist)

Seguridad — capas y controles
Capa 1: red, transporte y cabeceras HTTP

Los controles se agrupan en cuatro capas concéntricas: red, transporte, aplicación y datos. Un fallo en una capa no compromete las siguientes.

Capa de red y transporte

01
HTTPS forzado
Todo el tráfico circula sobre TLS 1.3. HTTP redirige a HTTPS por defecto.
02
HSTS preload
max-age=31536000; includeSubDomains; preload — el navegador rechaza HTTP futuro durante 1 año.
03
CDN distribuida
Mitigación implícita de DDoS volumétrico vía red global de Firebase.

Capa de aplicación · cabeceras

04
Content-Security-Policy
Política estricta: default-src 'self'; conexiones limitadas a Supabase/Stripe; frame-ancestors 'none' impide clickjacking.
05
X-Frame-Options DENY
Refuerzo doble anti-clickjacking para navegadores antiguos.
06
X-Content-Type-Options
nosniff — el navegador no interpreta MIME types incorrectos.
07
Referrer-Policy
strict-origin-when-cross-origin — no fuga de paths privados a terceros.
08
Permissions-Policy
Bloquea cámara, micrófono, geolocalización; permite payment sólo a Stripe.
09
Cross-Origin-Opener-Policy
same-origin — aísla la pestaña principal de pop-ups maliciosos.

Pagos · separación de responsabilidades

▣

Madcry Studio nunca almacena ni recibe datos de tarjeta. El proceso de cobro ocurre íntegramente en infraestructura de Stripe (PCI-DSS Level 1). Madcry sólo recibe del webhook el resultado del pago, los datos de envío del cliente y el identificador opaco cs_… de la sesión.

Gestión de secretos
Inventario y procedimiento de rotación

Las credenciales se clasifican en dos grupos según su exposición: las que viajan en el bundle del navegador y las que viven exclusivamente en infraestructura servidor.

Públicas (visibles en bundle del navegador)

Madcry Studio · Documentación técnica Gestión de secretos · 12 v1.0 · Mayo 2026

Reglas internas de manejo

  • Nunca pegar una credencial privada en chat, email, ticket o screenshot.
  • Nunca persistirla en .env.local salvo si es estrictamente PUBLIC_*.
  • Si es necesaria de forma puntual, exportarla en la sesión de terminal y borrar el histórico (history -c).
  • Rotar inmediatamente cualquier credencial que haya estado en un dispositivo perdido, repositorio público o canal compartido.
  • Toda rotación queda registrada con fecha en este documento (Anexos · Registro).
!

Política de mínimo necesario: el archivo .env.local que vive en la máquina de desarrollo contiene exclusivamente claves PUBLIC_*. Las claves potentes están en Supabase secrets — accesibles a la persona con login del proyecto, no al sistema de archivos del developer.

Operación, backups y continuidad
Despliegue, respaldo y recuperación

Despliegue

Cada commit en main dispara deploy.yml de GitHub Actions: npm ci, astro build y deploy a Firebase Hosting. Tiempo medio: 2-3 minutos. También se dispara desde el panel admin (función trigger-rebuild).

Backups

Imágenes (Storage)ContinuoIndefinidoReplicación interna
CódigoEn cada commitIndefinidoGitHub distribuido
Versión desplegadaEn cada build10 versionesFirebase rollback

Recuperación ante incidencias

EscenarioProcedimientoRTO objetivo
Bug en producciónRollback en Firebase a versión anterior< 5 min
Pérdida de datos en BDRestauración de backup + replay de webhooks< 4 h
Compromiso de credencialRotación + auditoría de logs< 1 h

Monitorización

Logs estructurados en cada Edge Function (Supabase Dashboard). Stripe envía email automático ante webhooks fallidos. Resend reporta entrega/rebote. Próxima iteración: integración con Sentry y agregación en una sola consola.
